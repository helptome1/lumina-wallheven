# Wallhaven Tauri UI 设计文档

本文档记录当前 `wallhaven-tauri` 版本的视觉风格、布局结构、组件样式和交互反馈规范。目标是让后续页面与组件调整保持一致，不让局部功能破坏整体气质。

## 1. 设计方向

当前 UI 方向可以概括为：轻量、通透、桌面端、毛玻璃、图片优先。

设计关键词：

- Light Glassmorphism：浅色毛玻璃界面。
- Image First：壁纸图片始终是视觉中心。
- Calm Desktop：弱化装饰，强调可浏览、可操作、可长期使用。
- Soft Depth：用透明度、模糊、阴影和边框制造层次。
- Consistent Feedback：收藏、删除、目录选择等操作使用同一套轻量反馈语言。

界面不走强营销风，也不使用重色块或大面积纯色背景。背景、侧栏、顶部栏、卡片和弹窗都应围绕“图片内容清晰可见，操作层轻轻浮上来”展开。

## 2. 布局结构

应用采用桌面端双栏结构：

- 左侧：固定宽度侧边栏，承载导航分组。
- 右侧：主内容区，承载顶部筛选栏、图片网格、详情弹窗和下载中心。
- 顶部：内容区内的玻璃质感标题/筛选区域。

布局原则：

- 主内容区保持充足留白，但不做营销式大 Hero。
- 图片列表优先使用响应式网格。
- 下载、收藏等管理页面使用更规整的列表和面板。
- 弹窗应覆盖内容，但保留下方页面的隐约可见性。

## 3. 视觉基调

### 背景

整体背景为浅色柔和底色，配合细微渐变和玻璃面板。背景不应抢走图片内容的注意力。

推荐特征：

- 浅灰、近白、轻微冷色调。
- 使用透明玻璃层承载导航和信息。
- 避免深色大面积覆盖图片。

### 主色

当前主操作色偏蓝紫，用于导航激活、主要按钮和重点状态。

辅助色：

- 橙色用于下载等行动性较强的按钮。
- 红色用于删除、错误等危险操作。
- 绿色可用于完成、成功等状态。

使用原则：

- 主色用于“当前所在位置”和“主要动作”。
- 危险色只在删除确认、错误反馈中使用。
- 不要把页面整体染成单一主色，图片内容需要保留主导权。

## 4. 字体与文字层级

当前界面使用偏现代的无衬线字体组合：

- 标题：更有展示感的无衬线字体。
- 正文：清晰易读的系统/Inter 类字体。
- 标签和数值：可使用更紧凑的字重与小字号。

文字原则：

- 浏览页标题简短，避免解释型长文案。
- 卡片信息使用小字号，不遮挡图片。
- 操作按钮尽量用图标表达，必要时用短文本辅助。
- 弹窗详情信息以数据分组展示，不写长段说明。

## 5. 侧边栏

侧边栏是应用信息架构的锚点。

当前结构：

- 顶部品牌区。
- 分组导航。
- 当前路由高亮。
- 使用毛玻璃背景和柔和边框。

交互规范：

- 当前菜单项需要有清晰的激活状态。
- Hover 状态可以轻微提亮或增加背景透明度。
- 菜单命名需与产品语义一致，不使用技术参数名。
- 新增菜单时应先确定所属分组，再配置路由与默认查询参数。

## 6. 顶部筛选栏

顶部区域承载当前页面标题、搜索和筛选控制。

设计原则：

- 筛选项保持紧凑，适合重复使用。
- 搜索输入应清晰可见，但不压过图片网格。
- 按钮、下拉和输入框使用统一玻璃/浅色控件风格。
- 筛选变化需要即时反映到图片列表。

## 7. 图片网格与卡片

图片卡片是应用的核心视觉单元。

卡片规范：

- 图片铺满卡片，保持良好裁切。
- 卡片圆角、阴影和边框提供轻微层次。
- 悬浮时显示操作层，但不能过度模糊图片。
- 操作层中的图标按钮应清晰、紧凑、易点击。

当前 hover 策略：

- 毛玻璃模糊强度较低。
- 用户仍可隐约识别小图内容。
- 信息和按钮浮在图片之上，但不遮挡主体过多。

建议避免：

- 大面积不透明遮罩。
- 过重模糊导致图片完全不可辨认。
- 在卡片内堆叠过多文字说明。

## 8. 大图详情弹窗

详情弹窗是从浏览到决策的关键场景，需要同时满足欣赏、查看信息和执行操作。

当前设计：

- 弹窗背景为透明毛玻璃覆盖层。
- 大图区域优先展示图片本身。
- 高清图未加载完成时，用缩略图过渡，避免空白等待。
- 右侧详情面板为透明玻璃质感，可隐约看到下方页面。
- 竖图使用完整包含显示，避免裁切。

交互规范：

- 关闭按钮始终可见。
- 放大按钮只改变图片查看状态，不改变弹窗布局。
- 放大后支持拖动查看图片细节。
- 关闭弹窗或切换图片时，应重置缩放和拖动状态。
- 图片加载失败时应回退到缩略图或给出可理解反馈。

## 9. 下载中心

下载中心属于管理型界面，相比浏览页应更稳、更清晰。

当前布局：

- 下载目录选择卡片。
- 下载中任务列表。
- 已完成下载记录列表。

设计规范：

- 标题卡片和内容卡片宽度保持一致。
- 下载目录选择使用和应用一致的玻璃面板风格。
- 下载中状态需要突出进度、速度和取消入口。
- 已完成记录左侧显示缩略图，右侧提供定位文件、删除等操作。
- 删除确认浮层应出现在删除按钮附近，而不是页面中央。

删除确认浮层：

- 使用小型浮层，减少打断。
- 文案直接询问是否删除本地图片。
- 保留取消和确认两个动作。
- 确认动作使用危险色。

## 10. Toast 反馈

Toast 用于轻量、即时、非阻断反馈。

适用场景：

- 收藏成功。
- 取消收藏。
- 操作失败。
- 下载或删除结果。

设计规范：

- 使用统一 `toast-glass` 视觉。
- 出现位置保持稳定。
- 文案短，直接说明结果。
- 不用于需要用户决策的场景；需要决策时使用确认浮层。

## 11. 图标与按钮

按钮应尽量使用图标表达明确动作：

| 动作 | 推荐表达 |
| --- | --- |
| 收藏 | Heart |
| 下载 | Download |
| 打开目录 | Folder |
| 定位文件 | FolderOpen |
| 删除 | Trash |
| 关闭 | X |
| 放大 | ZoomIn / ZoomOut |

规范：

- 图标按钮需要稳定尺寸，避免 hover 或状态变化造成布局抖动。
- 危险按钮只在删除、清理等场景使用。
- 主要按钮和次要按钮应通过颜色、背景和边框区分。
- 同一页面内同类按钮的尺寸和圆角保持一致。

## 12. 动效与状态

动效应轻，不应抢走图片注意力。

推荐状态：

- Hover：轻微上浮、提亮、显示操作层。
- Active：轻微压低或增强背景。
- Loading：使用骨架屏或缩略图过渡。
- Empty：使用统一空状态组件。
- Error：给出可读错误信息，并保留重试空间。

大图加载的特殊要求：

- 不显示空白大区域。
- 缩略图先出现，高清图完成后自然替换。
- 加载中可以叠加轻量状态，不遮挡整体结构。

## 13. 响应式原则

虽然当前应用以桌面端为主，但组件仍应具备基本响应能力。

原则：

- 网格列数根据宽度变化。
- 详情弹窗在较窄宽度下应避免信息面板挤压图片。
- 文本不能溢出按钮或卡片。
- 固定格式元素需要稳定尺寸，避免状态切换造成跳动。

## 14. 扩展建议

新增 UI 时优先复用以下风格：

- 玻璃面板：用于侧栏、顶部栏、弹窗、下载设置。
- 图片卡片：用于壁纸相关重复项。
- 小确认浮层：用于局部危险操作。
- Toast：用于操作结果反馈。
- 轻量图标按钮：用于高频操作。

需要避免：

- 新增完全不同风格的纯色大卡片。
- 过重阴影、过强模糊或过高不透明遮罩。
- 用长说明文字解释显而易见的功能。
- 在图片浏览场景中加入过多装饰性元素。
